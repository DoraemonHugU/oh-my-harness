# architect-design

## Overview

这个文件阐述了架构设计,功能设计,接口设计等设计相关的原则.

## 设计原则

### 状态机设计

设计一个状态机来管理系统的不同状态和状态之间的转换,确保系统在不同状态下的行为是可预测和一致的.
状态机起步设计不应该复杂化.
**对于大多数模块, 2-3个状态是完全可以进行表达的**

#### 状态机设计原则.

1. 当对一个新的模块设计状态机时,首先要明确模块的不同状态和状态之间的转换条件.
2. 拆分成明确的二到三个状态. 
3. 依据不同状态的表现形式: 首先确定明确的两个边缘状态,例如 "未登录" 和 "已登录". 这两个状态应该是系统的核心状态,并且具有明确的行为和响应. 
4. 如果有必要,则再确定一个模糊叠加状态,登录中.
5. 未来项目演进过程中,非特殊情况一般两端状态无需改变语义, 中间叠加状态由于多状态语义重叠,根据`状态机设计原则`对该状态进行递归设计,最终产出状态列表进行`flatten`处理,得到最终状态列表.

### DEV阶段开发

当项目处于DEV阶段时,非特殊必要情况,不应该考虑写兼容旧代码的逻辑(包含数据库迁移等).
DEV阶段非用户强调,则不存在任何有价值的数据.清理重建的工作量远小于兼容旧代码和迁移的工作量.
